# 로그인

## 개요

백오피스 최초 진입 시 표시되는 인증 화면입니다. 아이디·비밀번호 검증은 `POST /api/v1/auth/login`에서 수행하고, 2차 인증(이메일)이 켜진 사용자는 동일 세션으로 `POST /api/v1/auth/login/second-factor`까지 완료해야 세션이 확정됩니다. 세션 확인은 `GET /api/v1/auth/me`, 로그아웃은 `POST /api/v1/auth/logout`(인증 필요)입니다.

## 경로·구성

| 경로 | 설명 |
|------|------|
| `/` | 인증 여부에 따라 `/login` 또는 `/home`으로 이동 |
| `/login` | 로그인 카드 UI (이미 인증된 경우 `/home`으로 이동) |
| `/home` | 인증 필요. 미인증 시 `/login`으로 이동 |

## 화면 기능

- 아이디·비밀번호 입력 후 로그인 제출
- **2차 인증**: `user_info.second_auth_yn = 'Y'`인 계정은 비밀번호 검증 성공 후 인증번호 입력 단계로 전환. 등록 이메일(`email_id`)로 발송된 코드 입력 후 `로그인` 완료
- 비밀번호 표시/숨김 토글
- **아이디 기억하기**: 체크 후 로그인 성공 시 `localStorage`에 아이디를 **3일**간 보관, 다음 방문 시 아이디 입력란에 자동 반영. 미체크 후 성공 시 저장값 삭제
- **비밀번호 분실**: 셀프 재설정 없음. 로그인 카드 **왼쪽(히어로) 영역**에 안내 문구 표시 — 상세는 [비밀번호초기화.md](비밀번호초기화.md)
- 상단 언어 선택(ko/en, `localStorage` 유지)
- **로그아웃**: 인증 후 상단 레이아웃(`AppLayout`)에서 `POST /api/v1/auth/logout` 호출·세션 무효화 후 `/login`으로 이동
- 로그인 정책(백엔드)
  - 허용 등급: `ADMIN`, `MANAGER`
  - 사용자 상태: `ACTIVE`만 허용 (`LOCKED` 등은 차단)
  - `is_deleted = false`만 허용
  - **`access_ip_limit = 'Y'`**: 클라이언트 IPv4가 `user_info.access_ip`(단일 IPv4 또는 IPv4 CIDR 목록)에 포함될 때만 허용. 실패 시 `접속 가능한 IP가 아닙니다.` (코드 `ERR_ACCESS_IP_NOT_ALLOWED`, HTTP 403)
  - 비밀번호 연속 실패가 상한(`am.auth.max-password-fail-count`, 기본 5)에 도달하면 `password_fail_cnt` 증가와 동시에 **`user_status`를 `LOCKED`로 갱신**

## API

| 메서드·경로 | 용도 |
|-------------|------|
| `POST /api/v1/auth/login` | 아이디/비밀번호 검증. 2차 인증 불필요 시 세션 생성·완료 응답. 필요 시 `code: SECOND_FACTOR_REQUIRED`, `data.secondFactorRequired: true` (세션은 아직 미로그인, 인증번호는 서버 세션에 보관) |
| `POST /api/v1/auth/login/second-factor` | 본문 `{ "userId", "code" }`. 대기 중인 세션·코드 검증 후 세션 생성 |
| `POST /api/v1/auth/logout` | 인증된 세션만. HTTP 세션·SecurityContext 무효화 |
| `GET /api/v1/auth/me` | 세션 유무 판별. 성공 시 `data`에 `userId`, `displayName`, `gradeCd`(등급 코드) 포함. **HTTP는 항상 200**이며, 미인증은 `success: false`·`ERR_UNAUTHORIZED`로 구분 (브라우저 콘솔 401 노이즈 방지) |

### 로그인 응답 코드 참고

| code | HTTP | 설명 |
|------|------|------|
| `SUCCESS` | 200 | 로그인(또는 2차 인증) 완료 |
| `SECOND_FACTOR_REQUIRED` | 200 | 비밀번호까지 통과, 이메일 인증 대기 |
| `ERR_INVALID_CREDENTIALS` | 200 | 아이디 없음·비밀번호 불일치 등 (`success: false`) |
| `ERR_PASSWORD_FAIL_EXCEEDED` | 200 | 실패 횟수 상한(또는 이미 상한 상태) |
| `ERR_ACCESS_IP_NOT_ALLOWED` | 200 | IP 제한 위반 |
| `ERR_AUTH_GRADE_NOT_ALLOWED` | 200 | 등급 불가 |
| `ERR_USER_STATUS_NOT_ALLOWED` | 200 | 상태 불가 |
| `ERR_SECOND_AUTH_EMAIL_MISSING` | 200 | 2차 인증 on인데 `email_id` 없음 |
| `ERR_SECOND_FACTOR_INVALID` | 200 | 인증번호 불일치·만료·세션 불일치 |
| `ERR_UNAUTHORIZED` | 200 | `GET /auth/me` 만: 세션 없음·미인증 (`success: false`) |

## DB·데이터

- 테이블: `om_user_m`, 부가 정보는 `user_info` JSONB ([om_user_m.sql](../../ddl/om_user_m.sql) 참고)
- 로그인 검증에 쓰는 키 예: `password`, `grade_cd`, `user_status`, `password_fail_cnt`, `access_ip_limit`, `access_ip`, `second_auth_yn`, `email_id`, `last_login_dtm`

## 프로세스 흐름

### A. 아이디·비밀번호 (`OmAuthenticationProvider` → `AuthController`)

1. 아이디로 사용자 조회 (`WHERE user_id = ? AND is_deleted = false`)
2. 없으면 실패 (`사용자 정보가 일치하지 않습니다.`, `ERR_INVALID_CREDENTIALS`)
3. 있으면 순서대로 검증
   - **`access_ip_limit = 'Y'`** 이면 요청 IP(IPv4, `X-Forwarded-For` 첫 홉 우선)가 `access_ip` 규칙에 맞는지 검사. 실패 시 `접속 가능한 IP가 아닙니다.` (`ERR_ACCESS_IP_NOT_ALLOWED`)
   - 비밀번호 실패 횟수가 이미 상한 이상이면 `비밀번호 실패 횟수가 초과되었습니다.\n관리자에게 문의하세요.` (`ERR_PASSWORD_FAIL_EXCEEDED`)
   - 비밀번호 검증: 불일치 시 `password_fail_cnt` +1, **증가 후 값이 상한 이상이면 `user_status = LOCKED` 동시 반영**, 응답은 상한 시 위와 동일 메시지·그 외는 `사용자 정보가 일치하지 않습니다.`
   - 허용 등급(`ADMIN`, `MANAGER`), 사용자 상태(`ACTIVE`)
4. 위까지 통과 시 **아직** `last_login_dtm`·세션 확정은 하지 않음(다음 B/C에서 처리)

### B. 2차 인증 없음 (`second_auth_yn ≠ 'Y'`)

1. `AuthController`에서 세션에 `AuthSessionPrincipal` 저장
2. `last_login_dtm` 갱신 (`OmUserAuthMapper.updateLastLoginDtm`)

### C. 2차 인증 있음 (`second_auth_yn = 'Y'`)

1. `email_id` 없으면 `ERR_SECOND_AUTH_EMAIL_MISSING` (HTTP 200, `success: false`)
2. 서버 세션에 인증번호·만료 시각 저장, 이메일 발송(메일 미설정 시 서버 로그에 코드 출력 가능)
3. 클라이언트에 `SECOND_FACTOR_REQUIRED` 응답(세션 쿠키는 있으나 SecurityContext 미인증 상태 유지)
4. 사용자가 `POST /api/v1/auth/login/second-factor`로 코드 제출
5. 검증 성공 시 세션 저장 + `last_login_dtm` 갱신
